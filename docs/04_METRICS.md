# Метрики релизного продукта

## Принцип измерения

Метрики нужны для решений о продукте, а не для отчета количеством событий. Для каждого изменения заранее фиксируются:

- гипотеза;
- основная метрика;
- защитная метрика;
- период наблюдения;
- решение при положительном, нейтральном и отрицательном результате.

До получения первого стабильного baseline численные цели помечаются как `TODO`. Их нельзя назначать только по внешним отраслевым средним.

## Главная метрика

### Repeat Expedition Rate

Доля игроков, которые после первой завершенной или прерванной вылазки начинают еще одну вылазку в пределах окна наблюдения.

```text
Repeat Expedition Rate =
уникальные игроки с expedition_started >= 2
/
уникальные игроки с expedition_started >= 1
```

Основное окно для релиза: 7 календарных дней после первой вылазки.

Дополнительные срезы:

- повтор в той же сессии;
- повтор в течение 24 часов;
- повтор в течение 3 дней;
- повтор в течение 7 дней;
- Telegram и browser demo;
- результат первой вылазки;
- был ли PvP;
- было ли первое улучшение.

## Удержание

### D1 / D3 / D7 Retention

Игрок считается вернувшимся, если в соответствующий день совершил осмысленное действие, а не только открыл страницу.

Рекомендуемое qualifying action:

- `expedition_started`;
- либо `base_upgrade_completed`;
- либо `pvp_match_started`.

```text
Dn Retention =
игроки когорты первого входа с qualifying action в день N
/
все игроки когорты первого входа
```

Отдельно считать:

- новый зарегистрированный игрок;
- browser demo;
- вернувшийся существующий игрок;
- источник входа, если он известен.

## Продуктовые метрики

### First Upgrade Rate

Доля новых игроков, завершивших первое улучшение базы или снаряжения.

Окна:

- в первой сессии;
- до второй завершенной вылазки;
- в течение 24 часов.

Низкое значение означает одно из трех: цель непонятна, цена слишком высока или эффект не выглядит полезным.

### Risk Choice Rate

Доля случаев, когда игрок выбирает «идти глубже» среди событий, где доступны «идти глубже» и «вернуться».

```text
Risk Choice Rate =
risk_choice_made(choice=deeper)
/
risk_choice_presented
```

Считать по зоне, глубине, объему текущей добычи, состоянию игрока и предыдущему исходу.

Слишком низкое значение указывает на непривлекательную награду или чрезмерную угрозу. Слишком высокое может означать, что риск не ощущается.

### PvP Re-entry Rate

Доля игроков, которые после первой завершенной PvP-стычки начинают следующую PvP-стычку в течение 7 дней.

Срезы:

- победа / поражение / ничья;
- оружие;
- длительность боя;
- выход из вылазки после PvP;
- повтор через отдельный PvP-вход или событие `pvp_trace`.

### Content Exhaustion Time

Медианное время или число завершенных вылазок до момента, когда игрок увидел 80% доступного ему контента текущей версии.

Нужно считать:

- уникальные `content_id`;
- доступный набор с учетом открытых зон;
- повтор события;
- продолжения дневниковых цепочек.

Метрика показывает, когда требуется новый контент, но не заменяет оценку качества повторов.

### Weapon Pick Rate

```text
Weapon Pick Rate =
выборы оружия X
/
все подтвержденные действия раунда
```

Считать по уровню/рейтингу, сопернику, результату предыдущего раунда и режиму входа.

### Weapon Win Rate

```text
Weapon Win Rate =
победы в матчах, где оружие X было выбрано
/
матчи с выбором оружия X
```

Дополнительно нужен Round Damage Rate, потому что игрок может менять оружие по раундам. Одной матчевой win rate недостаточно для балансного решения.

### Resource Earn / Spend

Для каждого ресурса:

- выдано за день;
- потрачено за день;
- медианный баланс;
- доля игроков с нулевым балансом;
- основные источники;
- основные расходы;
- время до первого расхода;
- накопление без расхода.

Все операции должны иметь `reason`, `source_id` и баланс до/после.

## Воронка первой сессии

Обязательные шаги:

1. `app_opened`
2. `session_created`
3. `registration_completed`
4. `base_viewed`
5. `expedition_started`
6. `risk_choice_presented`
7. `risk_choice_made`
8. `expedition_finished`
9. `diary_entry_opened`
10. `upgrade_viewed`
11. `base_upgrade_completed`
12. `next_expedition_prompted`

Для каждого шага считать conversion и медианное время от предыдущего.

## Минимальная схема события

Каждое событие содержит:

```json
{
  "event_id": "uuid",
  "event_name": "expedition_started",
  "occurred_at": "2026-06-14T12:00:00Z",
  "player_id": "stable-id",
  "session_id": "session-id",
  "app_version": "0.9.0",
  "channel": "telegram",
  "properties": {
    "expedition_id": "uuid",
    "zone_id": "reed_edge",
    "attempt_number": 2
  }
}
```

Правила:

- `player_id` стабилен между сессиями;
- browser demo отделяется через `channel`;
- имена событий не меняются без миграции отчетов;
- персональные тексты сообщений не отправляются в продуктовую аналитику;
- сервер является источником истины для наград, расходов и исходов.

## Защитные метрики

Нельзя улучшать основную метрику ценой:

- роста ошибок входа;
- роста незавершенных вылазок из-за технического сбоя;
- увеличения времени загрузки;
- потери прогресса;
- ухудшения PvP Re-entry Rate;
- чрезмерного гринда до первого улучшения;
- роста обращений о непонятных последствиях.

## Производственные метрики малой команды

| Метрика | Как считать | Зачем |
| --- | --- | --- |
| Lead Time | от принятой задачи до production | Видеть реальную скорость доставки |
| Cycle Time | от начала разработки до готового изменения | Находить узкие места процесса |
| Deployment Frequency | успешные production-деплои за период | Проверять способность к live-работе |
| Change Failure Rate | доля деплоев с откатом или срочным исправлением | Контролировать качество |
| Mean Time to Restore | время от обнаружения до восстановления | Оценивать эксплуатационную готовность |
| Escaped Defects | дефекты, найденные после релиза | Улучшать QA, а не скрывать ошибки |
| Content Throughput | проверенные элементы контента за неделю | Планировать live-контент |
| Rework Rate | доля задач, возвращенных на существенную переделку | Находить неясные требования |

## Метрики AI/agent-assisted development

Измеряется не объем сгенерированного текста, а результат:

- доля задач с четким acceptance criteria до запуска агента;
- время от постановки до проверенного pull request/commit;
- доля AI-изменений, прошедших тесты с первого раза;
- число дефектов после AI-assisted изменения;
- доля кода/контента, потребовавшая существенной ручной переделки;
- время ручного review;
- повторное использование созданных тестов и инструментов;
- доля сессий агента с зафиксированным итогом и проверками;
- стоимость инструмента на выпущенное изменение.

Плохие метрики:

- количество сгенерированных строк;
- количество запросов к модели;
- процент «кода от AI» без учета качества;
- скорость черновика без времени на проверку.
