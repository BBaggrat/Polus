# Этап 1: техническая стабилизация

## Цель

Подготовить существующий продукт к безопасной реализации базы и вылазок. На этом этапе не добавлять контентную экспедиционную систему поверх неизвестных технических рисков.

## P0. Зафиксировать воспроизводимый baseline

Задачи:

- собрать проект чистой командой `mvn clean test`;
- запустить с файловой H2;
- проверить Telegram-вход и browser demo;
- пройти регистрацию, друзей, очередь, два хода боя, чат и завершение;
- записать текущие production URL, service name и deploy command;
- сохранить smoke-чеклист в репозитории.

Результат:

- документированный запуск;
- список известных дефектов;
- подтвержденная версия Java/Maven;
- повторяемая проверка перед каждой следующей задачей.

## P0. Аудит backend

Проверить:

- границы `controller → service → repository`;
- все endpoint и фактические DTO;
- соответствие `docs/API.md`;
- состояние после рестарта;
- thread safety матчмейкинга и дуэлей;
- таймауты и завершение зависших боев;
- browser demo isolation;
- обработку ошибок;
- JDBC initialization и будущие миграции.

Отдельное решение:

- какие сущности обязаны стать постоянными до 1.0: вылазка, ресурсы, апгрейды, дневник;
- нужна ли постоянность активной PvP-стычки или достаточно корректного восстановления/завершения.

## P0. Аудит frontend

Главная задача — стабилизировать `app.js`, не переписывая UI с нуля.

Порядок:

1. Определить реально активные функции после всех переопределений.
2. Добавить smoke-проверки на пользовательские сценарии.
3. Удалять дубли небольшими проверяемыми блоками.
4. Разделить минимум на модули состояния, API, home, social и duel.
5. Проверить обращения к отсутствующим endpoint, включая `/api/player/stats`.
6. Сохранить существующие DOM `id` до завершения стабилизации.
7. Проверить отсутствие mojibake в видимом тексте.

Результат:

- один активный путь инициализации;
- одна реализация каждого ключевого render/submit метода;
- отсутствие скрытых fallback-слоев, меняющих поведение по порядку объявления.

## P0. Аудит Telegram-бота

Проверить:

- `/start`;
- `getMe` и `deleteWebhook`;
- поведение при недоступном Telegram API;
- формирование Web App URL;
- cache-bust;
- логирование без токена;
- корректную остановку long polling;
- возможность отдельного health-сигнала для bot lifecycle.

Текст кнопки и приветствия пока можно оставить функциональным. Тематическое обновление выполняется в 0.9.

## P0. Проверка деплоя

Задачи:

- сверить `deploy/` с фактическим сервером;
- проверить владельца файлов и права `.env`;
- проверить `nginx -t`;
- проверить systemd restart policy;
- зафиксировать backup файловой БД;
- добавить команду отката на предыдущий JAR;
- выполнить production smoke после деплоя;
- проверить `/api/health` снаружи и локально;
- проверить свободное место и размер логов.

Результат:

- одна актуальная инструкция;
- deploy, rollback и backup проверены на практике;
- секреты не попадают в git и логи.

## P1. Логирование игровых событий

Расширить текущий `AppEventLogger` или подключить отдельное хранилище так, чтобы считать:

- `app_opened`;
- `session_created`;
- `registration_completed`;
- `matchmaking_joined`;
- `pvp_match_started`;
- `pvp_round_resolved`;
- `pvp_match_finished`;
- позднее: события вылазки, риска, ресурсов и апгрейдов.

Требования:

- стабильные имена;
- `player_id`, `session_id`, версия приложения;
- серверный источник истины для исходов;
- отсутствие текста приватных сообщений;
- экспорт, пригодный для расчета метрик.

## P1. Баланс в конфигурации

Без изменения текущих чисел:

- вынести HP, урон, снижение урона, шанс зацепа, награды и таймеры в типизированную конфигурацию;
- валидировать диапазоны при запуске;
- сохранить тесты текущей математики;
- логировать идентификатор версии баланса в PvP-событиях.

Не хранить критичные численные значения одновременно в Java и JavaScript.

## P1. Smoke-тесты

Минимальный автоматизируемый набор:

1. Healthcheck.
2. Browser demo session.
3. Регистрация.
4. Два независимых игрока.
5. Вход обоих в очередь.
6. Создание одной дуэли.
7. Отправка действий обоими игроками.
8. Обновление HP и лога.
9. Отправка сообщения.
10. Завершение и выдача результата.
11. Повторный вход после завершения.
12. Проверка 360, 390 и 430 px без горизонтального скролла.

Для Telegram отдельно оставить короткий ручной smoke до появления удобного интеграционного стенда.

## P1. Подготовка базы и вылазок

После стабилизации спроектировать, но не смешивать с PvP:

- `Expedition` как серверную сущность;
- состояния `PREPARING`, `ACTIVE`, `RETURNING`, `COMPLETED`, `FAILED`;
- `ExpeditionEvent` и выбранный outcome;
- журнал resource transaction;
- постоянные `BaseUpgrade`;
- связь события с `content_id`;
- идемпотентное подтверждение выбора;
- восстановление активной вылазки после обновления страницы и рестарта.

Перед кодом нужны:

- DTO и таблица состояний;
- минимальная схема данных;
- события аналитики;
- тесты переходов;
- решение по миграциям БД.

## Рекомендуемый порядок задач

1. Baseline и smoke-чеклист.
2. Инвентаризация активного frontend-слоя.
3. Контрактная сверка frontend ↔ backend.
4. Серверные тесты текущей PvP-математики и восстановления.
5. Проверка production deploy/rollback/backup.
6. Стабильная схема игровых событий.
7. Конфигурация баланса без изменения значений.
8. Автоматизированный smoke двух игроков.
9. Проектирование данных базы и вылазки.
10. Только после этого — реализация версии 0.9.

## Выходной критерий этапа 1

Этап завершен, когда:

- текущая игра запускается и проверяется одной инструкцией;
- активный frontend-код понятен и не зависит от цепочки случайных переопределений;
- frontend не вызывает несуществующие API в основных сценариях;
- деплой и откат проверены;
- ключевые PvP-события измеряются;
- баланс находится в одном валидируемом месте;
- smoke двух игроков повторяем;
- модель данных базы и вылазок согласована до реализации.
