# Производственная модель малой команды

## Формат команды

Sandalpunk / Полюс развивается как live-продукт малой команды. Команда не пытается имитировать крупную студию: роли совмещаются, но ответственность за решение всегда названа.

## Роли

### Основатель / разработчик

Отвечает за:

- продуктовое направление;
- приоритеты и scope;
- архитектурные решения;
- бизнес-логику и баланс;
- приемку изменений;
- production-деплой;
- работу с пользовательской обратной связью;
- решение о выпуске.

Не обязан вручную писать каждую строку, но остается владельцем результата.

### AI-агент разработки

Подходит для:

- аудита репозитория;
- локальных реализаций по acceptance criteria;
- тестов;
- миграций и конфигураций под review;
- поиска дублирования;
- технической документации;
- подготовки проверок и отчетов;
- воспроизводимых рутинных задач.

Каждая задача агенту должна содержать границы, запрещенные изменения и команды проверки.

### Нейросеть для текстов

Подходит для:

- вариантов событий;
- сокращения текста;
- поиска стилистических повторов;
- черновиков дневниковых записей;
- вариантов названий;
- проверки единого словаря.

Результат всегда проходит ручную редактуру и структурирование.

### Художник точечно

Подключается для элементов с высокой визуальной ценностью:

- ключевой арт;
- основная карта;
- набор направляющих по стилю;
- персонаж/силуэт;
- важные иконки и рамки;
- проверка целостности визуального языка.

Для повторяемых производных ассетов сначала создается утвержденный шаблон.

### QA / сайт-тестер

Отвечает за:

- проверку пользовательских сценариев;
- мобильные размеры;
- Telegram-клиенты и обычный браузер;
- два одновременных игрока;
- проверку текстов в реальном UI;
- фиксацию шагов воспроизведения;
- production smoke после деплоя.

Роль может выполнять привлеченный тестировщик, но автор изменения не должен быть единственным проверяющим крупного апдейта.

## Матрица ответственности

| Работа | Владелец решения | Исполнитель | Обязательная проверка |
| --- | --- | --- | --- |
| Product scope | Основатель | Основатель + AI для анализа | Сверка с vision и метриками |
| Архитектура | Основатель | Разработчик/AI | Review, тесты, миграционный план |
| Код | Основатель | Разработчик/AI | Автотесты и diff review |
| Баланс | Основатель | Разработчик + аналитика | Данные и закрытый тест |
| Тексты | Основатель/редактор | Нейросеть + человек | Ручная редактура |
| Арт | Основатель/арт-директор | Художник/инструмент | Проверка в интерфейсе |
| QA | Основатель | QA/сайт-тестер | Чеклист и отчет |
| Деплой | Основатель | Разработчик/AI | Healthcheck и production smoke |

## Что нельзя полностью делегировать AI

- окончательное продуктовое позиционирование;
- этические и юридические решения;
- решение о монетизации;
- оценку эмоционального качества мира;
- финальный баланс экономики и PvP;
- принятие риска потери данных;
- доступ к секретам без ограничений;
- production-деплой без проверяемого плана отката;
- приемку текста без чтения человеком;
- утверждение факта, что изменение готово к релизу.

AI может подготовить варианты и проверки, но ответственность остается у человека.

## Формат задачи

Каждая производственная задача содержит:

1. Пользовательскую проблему.
2. Измеримый результат.
3. Затрагиваемые файлы/системы.
4. Запрещенные изменения.
5. Acceptance criteria.
6. События аналитики.
7. Команды тестирования.
8. План отката для рискованных изменений.

Без этих полей задача сначала уточняется, а не сразу реализуется.

## Учет трудозатрат

Учитывается полное время изменения:

```text
подготовка требований
+ работа человека
+ работа AI-инструментов
+ review
+ тестирование
+ исправления
+ деплой
+ наблюдение после деплоя
```

Для планирования используется размер:

- **S:** до 1 рабочего дня, одна область, без миграции данных;
- **M:** 2–5 рабочих дней, несколько слоев или новый пользовательский сценарий;
- **L:** 1–3 недели, новая система, миграция, контент и аналитика;
- **XL:** задача должна быть разделена до начала.

Оценка включает резерв на QA и исправления. Сгенерированный AI-черновик не означает завершение задачи.

## Типы обновлений

### Малый апдейт

Состав:

- исправление;
- 1–3 контентных элемента;
- небольшая UI-полировка;
- настройка конфигурации без изменения модели данных.

Ориентир: несколько часов — 2 дня.

Проверка: целевой тест, smoke затронутого пути, healthcheck.

### Средний апдейт

Состав:

- законченный пользовательский сценарий;
- 4–12 контентных элементов;
- изменение нескольких frontend/backend-компонентов;
- обратимая миграция данных.

Ориентир: 3–7 рабочих дней.

Проверка: регрессия, мобильные размеры, staging/локальный production-like запуск.

### Крупный апдейт

Состав:

- новая зона;
- новая система прогрессии;
- существенная переработка основного цикла;
- крупный пакет контента.

Ориентир: 2–4 недели с разделением на промежуточные релизные шаги.

Проверка: отдельный scope, миграционный план, закрытый тест, измеримые критерии запуска.

## Ритм live-продукта

Рекомендуемый минимальный ритм:

- еженедельно: просмотр ошибок и ключевых метрик;
- раз в 1–2 недели: малый проверенный апдейт;
- раз в 4–8 недель: средний контентный или системный апдейт;
- после каждого деплоя: production smoke;
- после крупного изменения: короткий post-release review.

Ритм корректируется по фактической загрузке. Регулярность не должна достигаться ценой накопления дефектов.

## Критерий устойчивости модели

Производственная модель работает, если:

- один человек может понять текущее состояние по репозиторию и документам;
- изменение можно проверить без устных инструкций;
- контент не требует правки бизнес-логики;
- AI ускоряет путь до проверенного результата, а не увеличивает rework;
- production можно восстановить по документированной процедуре;
- roadmap обновляется по данным, а не по количеству идей.
