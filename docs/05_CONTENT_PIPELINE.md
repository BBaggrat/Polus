# Производство контента

## Цель

Контентный процесс должен позволять малой команде выпускать атмосферные события без неконтролируемого копирования текста в код. Источник истины — валидируемый JSON или таблица с устойчивыми идентификаторами.

## Типы событий

| Тип | Назначение |
| --- | --- |
| `loot` | находка ресурса, предмета, карты или полезного следа |
| `danger` | средовая угроза и решение о риске |
| `pvp_trace` | признаки другого игрока и переход к стычке или обходу |
| `monster` | встреча с существом или последствия его присутствия |
| `anomaly` | нарушение правил местности, обмен риска на знание/добычу |
| `quest` | шаг короткой цепочки с проверяемым состоянием |
| `diary` | находка или личная запись, раскрывающая мир и последствия |

## Структура события

Минимальная предлагаемая схема:

```json
{
  "id": "reed_edge_anomaly_001",
  "version": 1,
  "type": "anomaly",
  "zoneId": "reed_edge",
  "title": "Вода идет против ветра",
  "body": "Короткое наблюдение без скрытого результата.",
  "requirements": {
    "minDepth": 1,
    "requiredFlags": [],
    "excludedFlags": []
  },
  "choices": [
    {
      "id": "observe",
      "label": "Наблюдать",
      "riskHint": "средний",
      "outcomeId": "observe_result"
    },
    {
      "id": "leave",
      "label": "Обойти",
      "riskHint": "низкий",
      "outcomeId": "leave_result"
    }
  ],
  "outcomes": {
    "observe_result": {
      "text": "Результат решения.",
      "resourceChanges": [],
      "flagsAdded": ["saw_reverse_water"],
      "diaryEntryId": "diary_reverse_water"
    },
    "leave_result": {
      "text": "Игрок сохраняет добычу и меняет маршрут.",
      "resourceChanges": [],
      "flagsAdded": []
    }
  },
  "tags": ["water", "warning"],
  "analyticsKey": "reed_edge_anomaly_001"
}
```

Точная JSON-схема утверждается на этапе реализации. Обязательные свойства:

- уникальный неизменяемый `id`;
- версия;
- тип;
- зона;
- условия доступности;
- понятные варианты выбора;
- отдельные результаты;
- изменения состояния в структурированном виде;
- связь с дневником;
- ключ аналитики.

## Правила текста

- Ситуация до выбора не раскрывает скрытый результат.
- Кнопка описывает действие, а не настроение.
- Риск обозначается понятным сигналом, если персонаж способен его оценить.
- Один экран не содержит длинный рассказ и важное решение одновременно.
- Последствие объясняет причинно-следственную связь.
- Повторяемое событие имеет короткие варианты текста, но сохраняет одну механику.
- Имена мира и терминология сверяются с единым словарем.
- AI-черновик никогда не публикуется без ручной редактуры.

## Объем контента для 1.0

Минимальный набор — **48 текстовых элементов**:

| Тип | Количество |
| --- | ---: |
| `loot` | 8 |
| `danger` | 8 |
| `pvp_trace` | 6 |
| `monster` | 6 |
| `anomaly` | 8 |
| `quest` | 6 |
| `diary` | 6 |
| **Итого** | **48** |

Это минимум уникальных игровых элементов, а не количество отдельных строк результатов. Не менее 12 элементов должны поддерживать вариативный исход.

## Производственный процесс

```text
идея
  → AI-черновик
  → ручная редактура
  → внесение в JSON/таблицу
  → автоматическая валидация
  → функциональный тест
  → редакторский просмотр в интерфейсе
  → релиз
  → проверка метрик и обратной связи
```

### 1. Идея

Карточка идеи содержит:

- тип события;
- игровую функцию;
- ставку;
- возможные решения;
- связь с зоной;
- ожидаемое повторное использование;
- новую информацию о мире.

### 2. AI-черновик

Нейросеть может предложить:

- варианты формулировок;
- альтернативные исходы;
- короткие названия;
- проверку повторов;
- черновые теги.

В prompt передаются словарь мира, ограничения длины и игровая функция. Нельзя просить модель «просто написать атмосферное событие» без механической цели.

### 3. Ручная редактура

Ответственный человек проверяет:

- соответствие миру;
- ясность выбора;
- отсутствие ложных обещаний;
- длину на мобильном экране;
- причинность результата;
- тон;
- повторяемость;
- отсутствие случайных реальных культурных/географических утверждений.

### 4. Внесение в данные

Контент переносится в таблицу или JSON только после редактуры. Механические последствия задаются структурированными полями, а не извлекаются из текста.

### 5. Тест

Автоматически:

- уникальность `id`;
- допустимый `type`;
- существование ссылок на outcome и diary entry;
- допустимые resource/flag keys;
- наличие хотя бы одного выбора;
- ограничения длины.

Вручную:

- мобильное отображение;
- понятность без пояснения автора;
- корректность всех веток;
- соответствие награды риску;
- повторное появление.

### 6. Релиз

Каждый пакет получает:

- номер версии;
- список новых и измененных `content_id`;
- автора редакторского решения;
- дату;
- ожидаемую метрику;
- инструкцию отката или выключения проблемного события.

## Статусы элемента

```text
idea → drafted → edited → structured → validated → tested → released → retired
```

Нельзя выпускать статус `drafted` или `edited`, минуя структуру и тест.

## Контентный QA-чеклист

- [ ] Событие выполняет игровую функцию.
- [ ] Игрок понимает доступные действия.
- [ ] Риск не маскируется случайным текстом.
- [ ] Все outcomes достижимы.
- [ ] Награды и расходы используют существующие ключи.
- [ ] Дневниковая ссылка существует.
- [ ] Текст помещается на 360 px.
- [ ] Нет повторов с существующими событиями.
- [ ] Есть `analyticsKey`.
- [ ] Ручная редактура подтверждена.
