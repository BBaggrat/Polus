# Scope релизной версии 1.0

## Цель релиза

Версия 1.0 должна собрать существующую технологическую основу в цельную Telegram survival-RPG:

- игрок входит без технических препятствий;
- проходит вылазку;
- принимает хотя бы одно решение о риске;
- возвращает добычу;
- получает запись в дневник;
- видит и совершает первое улучшение;
- понимает, зачем выходить снова;
- может встретить другого игрока в формате текущей PvP-стычки.

## Системы 1.0

| Система | Минимальный состав для 1.0 | Критерий приемки |
| --- | --- | --- |
| База | Один главный экран убежища; ресурсы; 3–5 функциональных объектов или категорий улучшений; вход в карту, дневник и снаряжение | Игрок за 10 секунд понимает текущее состояние и следующий доступный шаг |
| Вылазки | Серверное состояние вылазки; старт, 3–5 событий, решение «глубже/вернуться», завершение, награда и последствия | Полный цикл проходит без ручного вмешательства и восстанавливается после обновления страницы |
| Дневник | Автоматические записи из событий; список; просмотр детали; минимум одна короткая цепочка | Запись связана с реальным действием и сохраняется между сессиями |
| PvP | Текущий матчмейкинг и duel engine; оружие, атака, уворот, чат, завершение; переименование в стычку в топи | Два игрока завершают бой, получают корректный результат и возвращаются в основной цикл |
| PvE | Текстовые встречи с существами/опасностями; 1–2 решения; последствия от подготовки и выбора | PvE не требует отдельной сложной боевой системы и дает понятный результат |
| Ресурсы | 3 базовых ресурса и 1 редкий класс; получение, расход, журнал транзакций | Баланс не создает отрицательные значения и позволяет сделать первое улучшение в первую или вторую сессию |
| Апгрейды | 6–8 улучшений в коротких ветках; стоимость; требования; видимый эффект | Первое улучшение доступно рано, эффект объяснен до покупки и заметен после нее |
| Карта | 3 зоны разной опасности; состояние открытия; выбор маршрута; краткие свойства зоны | Игрок видит разницу риска и причины открыть следующую зону |
| Аналитика | События входа, вылазки, риска, результата, улучшения, PvP и экономики; стабильный `player_id` | Можно посчитать метрики из `04_METRICS.md` без ручного чтения логов |
| Контент | 40–50 проверенных текстовых элементов; минимум 7 типов событий; таблица/JSON с версиями | Контент проходит валидацию схемы и ручную редактуру |
| QA | Smoke-набор; мобильные ширины; Telegram и браузер; проверка двух игроков; регрессия математики PvP | Блокирующие сценарии имеют повторяемую проверку перед деплоем |

## Что входит в 1.0

- новый визуальный и текстовый контекст аномальной топи;
- цельный цикл «база → вылазка → возврат → улучшение»;
- постоянное хранение критичного прогресса;
- адаптация существующего PvP под стычку в топи;
- небольшой PvE-слой на базе событий и решений;
- карта из нескольких зон;
- дневниковые записи, привязанные к действиям;
- базовая экономика без монетизации;
- серверная аналитика ключевых действий;
- браузерный демонстрационный режим;
- эксплуатационная документация и smoke-проверки;
- минимальный контентный набор для повторных вылазок.

## Что не входит в 1.0

- открытый мир с непрерывным перемещением;
- отдельная сложная тактическая PvE-боевая система;
- кланы, гильдии и войны объединений;
- торговая площадка между игроками;
- сложное производство и десятки рецептов;
- процедурная генерация текстов в production без ручной редакции;
- голосовой чат;
- нативные приложения iOS/Android;
- полноценные рейтинговые сезоны с призовым фондом;
- пользовательские карты и модификации;
- монетизация и pay-to-win-механики;
- рекламная сеть;
- масштабирование на несколько регионов до появления нагрузки.

## Границы баланса

Для 1.0 баланс должен обеспечивать:

- первое улучшение не позже второй завершенной вылазки;
- понятную цену решения «идти глубже»;
- отсутствие гарантированно лучшего оружия во всех PvP-сценариях;
- возможность восстановиться после неудачной вылазки;
- отсутствие бесконечного накопления без полезного расхода;
- отсутствие покупки преимущества за реальные деньги.

Численные значения утверждаются в версии 0.95 после появления событий аналитики и тестовых данных.

## Технические ограничения релиза

- существующий Spring Boot-монолит сохраняется;
- API изменяется эволюционно, без параллельной второй архитектуры;
- Telegram WebApp SDK остается основным каналом запуска;
- браузерная демо-сессия остается изолированной от постоянных Telegram-профилей;
- критичное состояние вылазки и прогресса не должно жить только в памяти процесса;
- текущая математика PvP не меняется на этапе 0;
- новые системы получают тестируемые сервисы и конфигурации, а не встраиваются копиями во фронтенд.

## Definition of Done для 1.0

Релиз готов, когда одновременно выполнены условия:

1. Новый игрок проходит полный первый цикл.
2. Повторная вылазка доступна без сброса аккаунта.
3. Основной прогресс переживает рестарт приложения.
4. Два реальных игрока могут завершить PvP-стычку.
5. Все обязательные продуктовые события записываются.
6. Контентный набор прошел ручную проверку.
7. Smoke-набор проходит на 360, 390 и 430 px, в Telegram и обычном браузере.
8. Нет блокирующих ошибок уровня запуска, входа, сохранения и завершения вылазки.
9. Есть процедура деплоя, отката и проверки health endpoint.
10. У команды есть список первых live-изменений после релиза.
