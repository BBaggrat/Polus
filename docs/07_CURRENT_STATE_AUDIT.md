# Аудит текущего состояния

Дата снимка: **14 июня 2026 года**.

Этот документ фиксирует существующую архитектуру перед переходом к релизной версии 1.0. Он не предлагает переписывание проекта.

## Репозиторий

- Один Maven-модуль.
- Spring Boot 3.3.5.
- Java 17.
- 77 Java-файлов в `src/main/java`.
- 3 тестовых класса в `src/test/java`.
- Статический frontend без React/Vite.
- Конфигурация деплоя для одного Linux VPS.

## Runtime-архитектура

Продукт является модульным монолитом в одном JVM:

1. Spring Boot отдает JSON API.
2. Spring Boot отдает статику Mini App по `/duel`.
3. Telegram-бот работает через long polling внутри того же процесса.
4. Nginx завершает HTTPS и проксирует запросы на порт приложения.
5. systemd запускает и перезапускает единый JAR.

Эта архитектура подходит для текущего масштаба и сохраняется на следующих этапах.

## Хранение

При конфигурации по умолчанию `APP_STORAGE=jdbc`:

- профили игроков хранятся через JDBC;
- друзья и заявки хранятся через JDBC;
- без внешнего `DB_URL` используется файловая H2 в `DB_DATA_DIR`;
- можно подключить PostgreSQL через переменные окружения.

Только в памяти процесса сейчас находятся:

- session tokens;
- матчмейкинг;
- активные дуэли и их чаты;
- браузерные демо-профили;
- последние 200 событий `AppEventLogger`.

Следствие: рестарт сохраняет профили и друзей, но сбрасывает сессии, очередь, активные бои, демо-профили и оперативную ленту событий.

## PvP

Текущий серверный бой:

- два игрока;
- 100 HP;
- одновременные действия;
- направления `LEFT`, `CENTER`, `RIGHT`;
- уворот `LEFT`, `STAY`, `RIGHT`;
- оружие `PISTOLS`, `RIFLE`, `SHOTGUN`;
- серверное разрешение раунда;
- чат и завершение боя;
- награды в монетах и рейтинге через `PlayerService`.

Математика зафиксирована в `DuelEngine` и покрыта отдельным `DuelEngineTest`. На этапе 0 она не меняется.

## Mini App

Frontend расположен в `src/main/resources/static/duel`:

- `index.html` — около 30 KB;
- `app.css` — около 136 KB;
- `app.js` — около 801 KB;
- `journal-events.js` — около 31 KB;
- локальные SVG-ассеты.

Реализованные пользовательские области:

- главный экран профиля;
- регистрация ника и стиля дневника;
- журнал;
- инвентарь;
- друзья и заявки;
- социальные чаты на клиентском уровне;
- магазинная оболочка;
- leaderboard на текущих/mock-данных;
- матчмейкинг;
- полный экран дуэли;
- браузерный сетевой demo flow.

Основной технический риск frontend: `app.js` содержит много повторных определений функций и несколько последовательно наложенных compatibility-слоев. Последний слой фактически переопределяет предыдущие реализации. Перед добавлением новых систем нужно определить активный слой и сократить дублирование под тестами.

## Telegram-бот

- Работает через Bot API long polling.
- Включается только при наличии `BOT_TOKEN` и `BOT_USERNAME`.
- Обрабатывает `/start`.
- Отправляет кнопку Web App на `${BASE_URL}/duel`.
- Работает в том же процессе, что backend.

Текст приветствия пока описывает вход в очередь на дуэль. В версии 0.9 его нужно привести к вылазке в топь.

## Тесты

Есть три серверных тестовых класса:

- `DuelEngineTest`;
- `MatchmakingServiceTest`;
- `PlayerServiceTest`.

Не обнаружены:

- автоматические browser smoke-тесты;
- тест Telegram `/start`;
- интеграционный тест JDBC-схемы;
- тест восстановления после рестарта;
- CI workflow в репозитории;
- контрактные тесты frontend ↔ backend.

## Чего нет для цикла 1.0

- серверной сущности вылазки;
- базы и постоянных улучшений;
- карты и открытия зон;
- серверной модели инвентаря;
- экономики ресурсов с журналом операций;
- контентной схемы событий;
- PvE-событий;
- постоянного дневника, связанного с результатами;
- продуктовой аналитики Repeat Expedition Rate;
- автоматизированного smoke-набора.

## Ограничения для следующих этапов

1. Не создавать рядом второй backend или новый frontend.
2. Сохранять модульный монолит до появления измеримой причины разделения.
3. Не добавлять системы поверх дублирующихся frontend-слоев без стабилизации.
4. Не хранить критичный прогресс вылазок только в памяти.
5. Не менять PvP-баланс без конфигурации, тестов и продуктовой гипотезы.
6. Не подключать монетизацию до живых метрик удержания.
