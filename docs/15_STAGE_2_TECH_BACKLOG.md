# Этап 2: технический backlog оболочки «Топь»

На этапе 1 задачи ниже не реализуются. Документ задает порядок проектирования следующей версии.

## Цель этапа 2

Добавить минимальный серверный и интерфейсный цикл:

```text
база → выбор зоны → вылазка → событие → возврат → сохранение результата
```

Текущий PvP сохраняется и встраивается как стычка в топи.

## P0. Модель состояния игрока

Задачи:

- определить постоянные ресурсы;
- добавить серверный инвентарь;
- добавить версию состояния;
- ввести журнал resource transaction;
- подготовить миграции БД;
- исключить критичные значения из `localStorage`.

Acceptance:

- состояние восстанавливается после рестарта;
- изменение ресурсов идемпотентно;
- отрицательный баланс невозможен;
- каждая операция имеет reason/source.

## P0. Модель вылазки

Предлагаемые состояния:

```text
PREPARING → ACTIVE → RETURNING → COMPLETED
                    ↘ FAILED
```

Задачи:

- `Expedition`;
- `ExpeditionEvent`;
- выбранный outcome;
- depth;
- carried loot;
- timestamps;
- optimistic version/idempotency key;
- восстановление активной вылазки.

Acceptance:

- одновременно активна не более одной вылазки;
- повторный запрос выбора не применяет награду дважды;
- обновление страницы не сбрасывает событие;
- рестарт приложения не теряет вылазку.

## P0. API вылазок

Черновые endpoint:

```text
POST /api/expeditions
GET  /api/expeditions/active
POST /api/expeditions/{id}/choices
POST /api/expeditions/{id}/return
GET  /api/zones
GET  /api/base
```

До реализации утвердить DTO, ошибки, идемпотентность и authorization.

## P0. Три зоны

Для каждой зоны нужны:

- стабильный `zoneId`;
- название и краткое описание;
- уровень опасности;
- требования открытия;
- таблица доступного контента;
- модификаторы риска/добычи;
- минимальный визуальный preview.

Названия зон — **TODO** после утверждения словаря мира.

## P1. Экран базы

Минимум:

- состояние игрока;
- основные ресурсы;
- вход в карту;
- активная/последняя вылазка;
- дневник;
- существующий профиль;
- вход в отдельную PvP-стычку при необходимости.

Не создавать параллельный frontend. Сначала сократить активный слой `app.js`, затем добавить screen с сохранением существующих DOM-контрактов.

## P1. Экран карты и старт вылазки

- три зоны;
- риск;
- ожидаемый тип находок;
- требования;
- выбор зоны;
- подтверждение старта.

Не раскрывать точные скрытые вероятности игроку, если это не часть предмета/улучшения.

## P1. Событийная структура

Типы:

- `loot`;
- `danger`;
- `pvp_trace`;
- `monster`;
- `anomaly`;
- `quest`;
- `diary`.

Нужны:

- JSON Schema;
- валидатор при сборке/запуске;
- stable content ID;
- version;
- requirements;
- choices;
- outcomes;
- analytics key.

## P1. Дневник

- серверные записи;
- связь с `contentId` и expedition;
- дата открытия;
- непрочитанное состояние;
- короткие цепочки;
- отдельный read endpoint.

Текущий клиентский журнал использовать как визуальный ориентир, но не как источник постоянного состояния.

## P1. PvP как стычка в топи

- переименовать тексты интерфейса;
- добавить контекст `pvp_trace`;
- передавать источник старта дуэли;
- определить ставку и результат для вылазки;
- сохранить `DuelEngine`;
- не менять баланс вместе с переименованием.

Отдельно решить:

- что происходит с вылазкой при рестарте во время PvP;
- какая добыча защищается/теряется;
- как исключить повторную выдачу награды.

## P1. Сохранение

Добавить миграции для:

- ресурсы;
- улучшения;
- зоны;
- вылазки;
- выбранные события;
- дневник;
- resource ledger.

До этого нельзя считать цикл релизным.

## P1. Аналитика

События из `04_METRICS.md`:

- `expedition_started`;
- `risk_choice_presented`;
- `risk_choice_made`;
- `expedition_finished`;
- `resource_earned`;
- `resource_spent`;
- `diary_entry_unlocked`;
- `base_upgrade_completed`;
- `zone_unlocked`.

Все серверные исходы содержат `expeditionId`, `zoneId`, `contentId`, `playerId`.

## P2. Контентный минимум

- подключить pipeline из `05_CONTENT_PIPELINE.md`;
- начать с 15 элементов для технической проверки;
- довести до 40–50 после стабилизации формата;
- отделить тексты от JavaScript и Java-кода.

## Порядок реализации

1. Схема данных и миграции.
2. Resource ledger.
3. Expedition state machine.
4. API и idempotency tests.
5. Три зоны.
6. 15 тестовых событий.
7. Экран базы и карты.
8. Восстановление состояния.
9. Дневник.
10. Встраивание PvP.
11. Analytics events.
12. Browser/Telegram smoke.

## Не входит в этап 2

- новая PvE-боевая система;
- монетизация;
- кланы;
- торговая площадка;
- процедурная генерация production-контента;
- микросервисное разделение;
- изменение PvP-баланса без отдельной задачи.

## Готовность этапа 2

- игрок проходит цикл базы и одной вылазки;
- есть три зоны;
- результат сохраняется после рестарта;
- события загружаются из структурированного источника;
- дневник получает запись из вылазки;
- PvP может быть запущен как контекстная стычка;
- Repeat Expedition Rate технически рассчитывается;
- smoke проходит в Telegram и браузере.
